# engineering-recap evals

Regression suite for the `bitwarden-daily-recap:engineering-recap` skill. Two
independent instruments cover the three ways this skill can regress:

| Surface                       | Files                                                    | What it catches                                                                                                                                                                                                                              |
| ----------------------------- | -------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triggering**                | `trigger-eval.json`, `run_real_eval.py`, `baseline.json` | The skill stops firing on natural-language recap requests, or starts firing on near-misses that belong to a sibling skill (`claude-retrospective`, PR review/status, meeting recaps).                                                        |
| **Behavior + output quality** | `evals.json`                                             | The produced HTML loses required structure (5-stat strip, standup card, themed streams, timeline, theme block), or the skill's logic regresses (day-boundary folding, cross-repo sweep deduplication, PR-title enrichment, output location). |

Run the **triggering** suite before merging any change to the skill's
`description` frontmatter. Run the **behavior** suite before merging any change
to the workflow steps, the bundled `template.html`, `gather-gh-events.sh`, or
`render-guide.md`.

## Triggering suite

### Why a custom runner

The upstream `skill-creator` harness measures triggering by registering a
temporary copy of the skill under a UUID-suffixed name and watching whether the
model invokes that exact name. When the real plugin-registered skill is already
installed in the test environment, the model invokes the real one and the
harness records a false negative. `run_real_eval.py` instead watches `claude -p`
stream events for any invocation of the real `engineering-recap` skill, ignoring
unrelated session-init or workflow skills that may fire first.

### Files

- `trigger-eval.json` — 17-query test set: 8 should-trigger natural-language
  phrasings (yesterday/this-week recaps, standup pressure, 1:1 prep,
  "interactive html page") and 9 should-not-trigger near-misses against sibling
  skills (`claude-retrospective` session inspection, PR review/status,
  commit-SHA lookups, Jira, repo-level overviews). Every query is deterministic
  — it reliably fires or reliably stays silent — so a non-empty diff against the
  baseline is a real regression, not sampling noise.
- `run_real_eval.py` — runner. Spawns parallel `claude -p` subprocesses, parses
  streamed tool-use events, computes per-query trigger rates.
- `baseline.json` — last known-good run. Diff against this to spot regressions.
  Generated by the first run in an environment where the `bitwarden-daily-recap`
  plugin is **installed and enabled** (so `engineering-recap` is a registered
  skill). Note: if the legacy personal `~/.claude/skills/daily-recap` skill is
  also installed, it shadows the plugin skill and skews triggering — retire the
  duplicate before measuring.

### Running

Requires Python 3.10+ and an authenticated `claude` CLI on `PATH`.

```bash
python3 run_real_eval.py \
  --eval-set trigger-eval.json \
  --runs-per-query 3 \
  --num-workers 8 \
  --timeout 60 \
  --model claude-opus-4-8 \
  > result.json
```

17 queries × 3 runs = 51 `claude -p` invocations. With 8 workers the run takes a
few minutes. Use the model ID that powers the environment the skill actually
runs in, so the triggering test matches real usage.

### Regression check

```bash
diff <(jq -S . baseline.json) <(jq -S . result.json)
```

Empty diff means no regression. If a new failure appears, fix the skill
description rather than the eval set — the eval set encodes intent, not
implementation. If the change is intentional and the new run is the desired
behavior, replace `baseline.json` with `result.json` and commit it alongside the
description change.

### Updating the test surface

Update `trigger-eval.json` (not the runner) when the test surface needs to
evolve: a new natural-language phrasing the skill should catch, a new sibling
skill creating a new near-miss, or an existing query that turned out to be
ambiguous. Keep should-trigger and should-not-trigger counts roughly balanced.

## Behavior + output-quality suite

`evals.json` follows the `skill-creator` schema: each entry has a `prompt`, an
`expected_output`, and a list of objectively gradable `assertions`. The
assertions deliberately test **structure and logic** (output location, the
5-stat strip, themed-stream grouping, day-boundary folding, sweep
deduplication, PR-title enrichment) rather than specific PR numbers or dates — the GitHub
events API only returns ~90 days / ~300 events, so any assertion pinned to a
specific historical PR would rot. Relative day references ("yesterday", "this
morning") are resolved against the run date; the explicit-date scenario uses
"the day before yesterday" to exercise non-default window selection without
hardcoding a perishable date.

### Running

Run on a machine with real Claude Code session history and GitHub activity in
the requested window (otherwise there is nothing to render). Use the
`skill-creator` skill's harness to execute each prompt with and without the
skill, grade the assertions, and produce a benchmark:

1. Invoke the `skill-creator` skill and point it at this skill and `evals.json`.
2. It spawns with-skill and baseline runs per scenario, grades each assertion,
   and aggregates a `benchmark.json` / `benchmark.md` with pass rates and the
   with-skill vs baseline delta.
3. Review the rendered HTML outputs in the eval viewer before accepting a change.

A healthy result is a high with-skill pass rate and a large positive delta over
the no-skill baseline (the original home-skill baseline measured 97% with-skill
vs 11% without — the skill is doing real work). Investigate any assertion that
passes equally with and without the skill: it is non-discriminating and should
be tightened or dropped.

### Updating the scenarios

The four scenarios each isolate a distinct behavior dimension — full-structure
recap, standup-card focus under time pressure, explicit-date window selection,
and partial/light-data honesty. Add a scenario when a new behavior dimension
needs coverage; keep assertions objective and decoupled from perishable data.
